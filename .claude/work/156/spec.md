# Spec ŌĆö #156: arch: collapse dual notification systems onto @hamak/notification

## Goal

Replace the bespoke `SimpleNotificationService` in `frontend/src/plugins/notification/notificationPlugin.ts` with the framework-provided `@hamak/notification` plugin factory so there is exactly one notification surface. The ticket states the problem as "Two parallel notification systems existŌĆ” Either the framework package is dead weight or this plugin is duplicating it. Two truths." Audit confirms the duplication is the dead-weight case: `@hamak/notification@^0.5.2` is declared in `frontend/package.json:22` but never imported, and the bespoke service has **zero external consumers** (grep over `frontend/src` for `NOTIFICATION_TOKEN`, `SimpleNotificationService`, `commands.execute('notification.*')`, and the bespoke `Notification`/`NotificationType` types returns hits only inside `notificationPlugin.ts` itself, and zero grep hits inside `frontend/src/test/` or any `__tests__/` directory). Cycle-1 review revealed three runtime side effects in `frontend/node_modules/@hamak/notification/dist/impl/plugin/notification-plugin-factory.js` that the cycle-1 spec missed (mandatory `LOG_MANAGER_TOKEN` resolve at `initialize`, auto-registered `notifications` Redux slice via `STORE_EXTENSIONS_TOKEN`, welcome-notification side effect when `maxNotifications !== undefined`). This cycle-2 spec bakes in user-resolved scope decisions for all three: add the logging plugin as a sibling, accept the slice, suppress the welcome notification. The framework package was also refactored to a single `@hamak/notification` with subpath exports `.`, `./api`, `./spi`; CLAUDE.md line 121 is stale and gets corrected here.

## Public surface (signatures)

### `frontend/src/plugins/notification/notificationPlugin.ts`

```ts
// frontend/src/plugins/notification/notificationPlugin.ts
//
// Thin adapter onto @hamak/notification. This plugin owns no notification
// state of its own ŌĆö it delegates the whole concern to the framework
// factory and only fixes app-level configuration here.

import { createNotificationPlugin as createFrameworkNotificationPlugin } from '@hamak/notification';
import type { PluginModule } from '@hamak/microkernel-spi';
import type { NotificationPluginConfig } from '@hamak/notification/spi';

/**
 * App-level configuration for the notification plugin.
 *
 * Tuned for smart-data-dico's UI: top-right placement, 5s default
 * auto-dismiss, no localStorage persistence (notifications are
 * session-scoped feedback for save/commit/import flows).
 *
 * `maxNotifications` is intentionally omitted: setting it to any value
 * triggers a welcome notification on activate (framework behavior at
 * `frontend/node_modules/@hamak/notification/dist/impl/plugin/
 * notification-plugin-factory.js` lines 103-109 ŌĆö
 * `if (config.maxNotifications !== undefined) { service.info(
 *   'Notification system ready', 'System', { duration: 3000, ŌĆ” }); }`).
 * We don't want that surfaced when the toast renderer lands in a
 * follow-up ticket. Framework default for `maxNotifications` is 50
 * (`spi/plugin/plugin-config.d.ts:17`), which is what we want anyway.
 */
const APP_NOTIFICATION_CONFIG: NotificationPluginConfig = {
  defaultDuration: 5000,
  position: 'top-right',
  enablePersistence: false,
};

/**
 * Factory used by `frontend/src/kernel/bootstrap.ts`.
 *
 * The returned `PluginModule` will, on `initialize`:
 *   1. `ctx.resolve(LOG_MANAGER_TOKEN)` ŌĆö REQUIRES the logging plugin to
 *      have initialized first (see `bootstrap.ts` registration order).
 *   2. Provide `NOTIFICATION_SERVICE_TOKEN` via DI.
 *   3. Best-effort register `notificationReducer` under `state.notifications`
 *      via `Symbol.for('@hamak/ui-store:StoreExtensionsRegistry')` (which
 *      ui-store does provide ŌĆö the slice WILL appear in `RootState`).
 *   4. Register the framework's seven `notification.*` commands with
 *      `(args: object)` signatures (NOT the legacy `(message: string)` form).
 *
 * On `activate`: emits `notification:ready` hook; does NOT emit a welcome
 * notification (suppressed by omitting `maxNotifications` above).
 */
export function createNotificationPlugin(): PluginModule {
  return createFrameworkNotificationPlugin(APP_NOTIFICATION_CONFIG);
}
```

### Framework command surface (canonical)

The framework's `createNotificationPlugin` registers these seven commands inside its `initialize` (`notification-plugin-factory.js:57-83`). They are the canonical public surface as of this ticket:

| Command | Args shape | Action |
|---|---|---|
| `notification.show` | `{ type?: 'info'\|'success'\|'warning'\|'error', message: string, title?: string, ...NotificationOptions }` | calls `service.notify(...)` |
| `notification.info` | `{ message: string, title?: string, ...NotificationOptions }` | calls `service.info(message, title, options)` |
| `notification.success` | `{ message: string, title?: string, ...NotificationOptions }` | calls `service.success(message, title, options)` |
| `notification.warning` | `{ message: string, title?: string, ...NotificationOptions }` | calls `service.warning(message, title, options)` |
| `notification.error` | `{ message: string, title?: string, ...NotificationOptions }` | calls `service.error(message, title, options)` |
| `notification.dismiss` | `{ id: string }` | calls `service.dismiss(id)` |
| `notification.dismissAll` | `(no args)` | calls `service.dismissAll()` |

Argument shape changed from the bespoke `(message: string)` to `(args: { message, title?, ŌĆ”options })`. Audit (grep over `frontend/src` for `commands.execute.*notification` and `notification\.\(success\|error\|warning\|info\)\b`) confirms **zero current call sites** depend on the old shape, so impact today is zero. Future callers consume the framework signature directly.

## Plugin load order

`bootstrap.ts` plugin registration order, post-edit:

```
[store, shell, auth, data-dictionary, visualization, search,
 version-control, remote-fs, remote-git,
 LOGGING,                       // ŌåÉ NEW: must be before notification
 notification,                  // ŌåÉ now depends on ['store', 'logging']
 cases, rules]
```

Rationale: the notification plugin's `initialize` does `ctx.resolve(LOG_MANAGER_TOKEN)` unconditionally (`frontend/node_modules/@hamak/notification/dist/impl/plugin/notification-plugin-factory.js:33`). The microkernel DI container at `frontend/node_modules/@hamak/microkernel-impl/dist/runtime/di.js:63` throws `No provider for token: ...` when a token has not been provided. The logging plugin provides `LOG_MANAGER_TOKEN` inside its own `initialize` (`frontend/node_modules/@hamak/logging/dist/impl/plugin/logging-plugin-factory.js:64`), so the logging plugin must initialize first.

Two layers of enforcement, belt-and-braces:

1. **Registration order**: logging is registered before notification in `bootstrap.ts` (the host generally initializes in registration order absent overriding `dependsOn`).
2. **Manifest `dependsOn`**: the notification plugin's manifest gains `'logging'` in `dependsOn`. This makes the constraint explicit and survives any future reordering of `registerPlugin` calls.

       host.registerPlugin('logging',      { name: 'logging',      version: '1.0.0', entry: '' },                                createLoggingPlugin());
       host.registerPlugin('notification', { name: 'notification', version: '1.0.0', entry: '', dependsOn: ['logging'] },        createNotificationPlugin());
       await host.bootstrapAllAtRoot();

       const svc = host.rootActivationCtx!.resolve<INotificationService>(NOTIFICATION_SERVICE_TOKEN);
       const listener = vi.fn();

       expect(all[0].duration).toBe(5000); // matches APP_NOTIFICATION_CONFIG.defaultDuration

## Risks

Cycle-1 review reassessed overall risk as **Medium** (was Low in cycle-1 spec). This cycle adopts that assessment.

1. **Boot-failure-without-logging.** If a future refactor accidentally removes the logging plugin registration from `bootstrap.ts`, or if a new top-level plugin is added before logging that itself resolves `LOG_MANAGER_TOKEN`, app boot will throw `No provider for token: ...` from `@hamak/microkernel-impl/dist/runtime/di.js:63`. Mitigation: belt-and-braces enforcement via both registration order AND `dependsOn: ['logging']` on the notification manifest. Plus Acceptance #6's bootstrap test fails fast if the order regresses.

2. **Cookbook does not cover "consuming a framework plugin factory."** The pattern this spec uses (`frontend/src/plugins/<name>/<name>Plugin.ts` re-exports a thin factory that delegates to the framework while fixing app-level config in a local `const`) is not in `frontend/docs/patterns.md`. Mitigation: once this ticket lands, add a worked example to `patterns.md` as a new ┬¦6 or as subsection ┬¦3a "Delegating to a framework-provided service." That cookbook update is a follow-up edit, not part of this spec; the example will be the merged form of `notificationPlugin.ts` so it stays anchored to real code.

3. **Hidden contract on `STORE_EXTENSIONS_TOKEN`.** The framework auto-registers `notificationReducer` under `state.notifications` if and only if `Symbol.for('@hamak/ui-store:StoreExtensionsRegistry')` resolves. In smart-data-dico that token is provided by `@hamak/ui-store-impl/dist/plugin/store-plugin-factory.js:91` (verified by cycle-1 reviewer), so the slice WILL appear. Mitigation: Acceptance #9's grep confirms no exhaustive-shape tests assert otherwise today. If a future test asserts an exhaustive `RootState` shape, it must add `notifications` to its expected keys.

4. **Command argument shape change `(message: string)` ŌåÆ `(args: { message, ŌĆ” })`.** No mitigation needed today (zero callers per audit), but documented in the framework-command-surface table above so future implementors do not pass a bare string. If a follow-up needs the legacy form for backwards compatibility, it adds a wrapper command under a new cookbook-shaped name (e.g. `notification.toast.success`); this spec does not pre-emptively add one.

5. **CLAUDE.md edit could silently desync again if the framework re-renames packages.** The framework already churned once (`-api` + `-impl` ŌåÆ single package). No mechanical mitigation. Acceptance #10's grep fails CI if `CLAUDE.md` and `frontend/package.json` drift again on these two specific packages. Future framework renames are a future ticket.
